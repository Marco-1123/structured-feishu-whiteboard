# Quality Checklist

生成飞书画板前后都要检查。不要只依赖命令成功。

## 信息质量

- 容量、指标和占位符判断以 `content-budget.md` 为准；本清单只做最终核对。
- 标题是否直接说明主题或结论。
- 是否有一个明确主结论。
- 是否只保留 3 到 6 个关键模块。
- 每个模块是否有独立信息价值。
- 是否删除了重复、铺垫和过程性说明。
- 是否避免把“用户让我整理”“来源是某文档”“使用某风格”等元信息写到画板上。
- 是否按 `content-budget.md` 判断了短内容、中等内容、长文或复杂材料。
- 长文是否先形成信息保全清单，再组织成顶部总览和同页后续区域，而不是把全文搬上画板。
- 是否清理了乱码、异常符号、长 URL、脚注和格式残片。
- 指标是否符合 `content-budget.md`：有业务语义、没有同类重复、呈现层级唯一。
- 指标区是否统一对齐：同宽、同基线、同字号、同间距。
- 单张画板是否只承载一个主表达任务；总览、指标、路线、对比是否被混在一起。

## 结构质量

- 版式是否符合内容关系。
- 阅读顺序是否明显。
- 重要信息是否在视觉上更突出。
- 同级模块是否尺寸和层级一致。
- 对比矩阵是否维度清楚。
- 路线图和流程图是否方向明确。
- 内容是否超过所选版式的容量边界。
- 如果内容超载，是否压缩或拆图，而不是缩小字号硬塞。
- 主结论是否位于主视觉区，而不是右上角补丁式小框。
- 并列模块是否误用了箭头；只有推进、流程或依赖关系才使用箭头。

## 视觉质量

- 颜色是否只使用选定风格库。
- 强调色是否克制。
- 卡片边距、行距和安全边距是否充足。
- 同一容器内的上下留白是否均衡；标题、正文、标签不能整体偏上或偏下。
- 卡片底部标签、按钮或徽标是否离底边至少保留一行文字高度的安全距离。
- 字号层级是否清楚。
- 是否存在无意义装饰。
- 是否避免过多边框、阴影和标签。
- 是否存在三层以上框套框；同一区域最多两层容器。
- 卡片内是否同时堆了动作列表、多个指标框和读法说明；如有，必须拆分。
- 是否存在上下两个指标层导致视觉重复和不对齐。
- 底部说明条是否抢走主内容注意力；专业蓝白风默认不用大面积黑色说明条。
- 箭头是否连接了明确对象；不能压住文字，也不能漂浮在两段文字之间造成读向歧义。
- 长文是否通过同页区域扩展增加容量，而不是生成难读的单个长竖图或切成多个页面。

## SVG 质量

- 正文使用 `<text>` / `<tspan>`。
- 主体结构使用 rect、circle、ellipse、line、polyline。
- 没有 gradient、filter、pattern、clipPath、mask。
- 没有透明度属性。
- 没有手绘箭头头部。
- `<path>` 只出现在 marker 里。
- 长文本已拆成明确 `<tspan>` 行，不依赖自动换行。

## 渲染检查

运行：

```bash
npx -y @larksuite/whiteboard-cli@^0.2.11 -i diagram.svg -o diagram.png -f svg
npx -y @larksuite/whiteboard-cli@^0.2.11 -i diagram.svg -f svg --check
node scripts/check-svg-layout.mjs diagram.svg
grep -nE '<polygon|opacity=|fill-opacity=|stroke-opacity=|<filter|<linearGradient|<radialGradient|<clipPath|<mask' diagram.svg
```

必须人工查看渲染图，重点检查：

- 文字是否溢出卡片。
- 模块是否互相重叠。
- 内容是否被画布裁切。
- 卡片是否太挤。
- 卡片内上下留白是否视觉均衡。
- 箭头是否与步骤、阶段或流程节点对齐。
- 主结论是否足够醒目。
- 是否出现乱码、控制字符、未转义符号或原文格式残片。
- 是否存在“看似没溢出但整体过密”的拥挤问题。
- 是否存在容器过密、边框过多导致的“框套框”观感。
- 子元素是否越过父容器边界；白板工具可能不会稳定发现这类问题，必须用本地布局检查和人工预览同时确认。

发现局部问题时，直接小范围修改 SVG：移动、加宽、换行、增高、删减文字。不要为了一个局部问题重写整张图。

如果局部修改仍需要降低到字号底线以下，按 `overflow-repair.md` 拆图。

## 交付检查

交付前确认：

- 飞书文档链接可用。
- 画板内容已写入，不是空白。
- 返回了预览图。
- 回复中说明采用的版式和风格。
- 回复中说明必要的信息压缩，但不冗长解释制作过程。
